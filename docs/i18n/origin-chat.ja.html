Plan-Gated Paper Revision Skill

AI を利用した原稿改訂を管理し、追跡可能にし、混乱を軽減する方法を説明する、クリーンで軽く脚色されたオリジン チャット。

一言で言えばアイデア

AI エージェントに一度に考え、編集、検証、説明をさせないでください。 まず推論を Markdown 計画に強制的に組み込んでから、編集エージェントにその承認された計画のみを実行させます。

1

まず考えてください

推論モデルを使用してレビュー担当者の懸念を診断し、計画を作成します。

2

ゲート編集

計画も原稿の変更もありません。プランは編集境界です。

3

真実を検証する

PDF をコンパイルして検査し、確認された場所で応答レターを更新します。

4

コミット履歴

受け入れられた各作業パッケージは、レビューまたはロールバックできるクリーンな Git ステップになります。

以前は AI を使用して論文を改訂するワークフローを使用していました。重要なのは、プロセスを直線的に保つことです。そうしないと、リビジョン全体が乱雑になってしまいます。

まず、提出されたバージョンを次のように保持します。 original_main.tex。作業ファイルはまだ呼び出されています main.tex。次に、エージェントに生成させます redlined_main.tex。レッドラインを提出していない場合でも、何が変わったのかを確認するのに非常に便利です。

オペレーティング システムとしてのワークフロー

この方法が機能するのは、乱雑なリビジョンを一連の小さな検証可能なトランザクションに変換するためです。

0

ベースラインを凍結する

キープする original_main.tex 変わらない。取り組む main.tex。レッドラインを自動的に生成します。

1

応答ロジックの草案を作成する

原稿に触れる前に、査読者の本当の懸念を理解してください。

2

references/ の下にプランを作成します

この計画では、許可されるファイル、許可される場所、禁止される変更、根拠、検証が定義されます。

3

人間が承認するか分裂するか

計画が大きすぎる場合は分割してください。明確であれば実行してください。

4

執行者は計画のみを適用します

Codex は、許可されたファイルを編集し、実行メモを記録します。

5

PDF の構築と検査

原稿と返答レターを作成します。ページまたは行の位置を要求する前に、実際の PDF を確認してください。

6

応答レターを同期する

返答レターには、計画された変更や想像上の変更ではなく、実際の原稿の変更が反映されている必要があります。

7

コミットしてプッシュする

受け入れられたすべての作業パッケージは、クリーンな Git チェックポイントになります。

reviewer comment
→ response logic
→ references/reviewer1_comment1_revision1.md
→ human approval
→ edit main.tex only within the plan
→ compile main.pdf + response_letter.pdf
→ verify PDF locations
→ update response letter
→ generate redline
→ git commit + push

エージェントの役割

1 人のエージェントがすべての役割を実行できますが、役割を分離するとワークフローがより明確になります。

戦略エージェント

レビュー担当者の懸念を診断し、計画を作成し、トーンを制御し、タスクを分割するかどうかを決定します。原稿を直接編集してはいけません。

執行者エージェント

承認された計画を読み取り、ファイルを編集し、LaTeX をコンパイルし、レッドラインを生成し、実行内容を記録します。科学を自由に再解釈すべきではありません。

監査代理人

PDF をチェックし、ページ/行参照を検証し、応答レターの真実性をチェックし、編集が計画を超えていないことを確認します。

人間の司令官

計画を承認し、タスクの粒度を決定し、科学的方向性を判断し、最終改訂を受け入れるか拒否します。

認知負荷の分割

よくある失敗プランゲート型ソリューション実践的なルール
エージェントが変わりすぎる計画では、許可されるファイルと場所を正確に定義します。計画にない場合は編集しないでください。
応答レターの超過請求場所の主張は、PDF 検査後にのみ追加されます。原稿が実際に修正される前に「修正しました」とは決して書かないでください。
更新履歴が不明瞭になる各作業パッケージにはプランと Git コミットがあります。1 つの計画、1 つの実行、1 つのコミット。
大きなタスクがエージェントに過負荷をかける大まかな計画は実行前に分割されます。曖昧な計画→分割計画。
新しいセッションはコンテキストを失いますの references/ ディレクトリは外部メモリとして機能します。編集する前に、エージェントに関連するプランを読んでもらいます。

2 つのリビジョン モードの比較

標準 Codex 命令

このプロンプトは、チャットの哲学を実行可能なエージェントの指示に変えます。

You are working on an academic manuscript revision.

First read the relevant plan in references/.
Do not edit anything outside the plan.
If the plan is too broad or vague, stop and propose a smaller plan.
Apply only the approved edits.
After editing, update the plan's Execution Notes.
Run the build command.
Inspect the generated PDF if available.
Update the response letter only with verified changes.
Generate redlined_main.tex when original_main.tex is available.
Show git diff summary.
Commit and push this work package.

言わないでください

「査読者のコメントに従って論文を修正してください。」

これは範囲が広すぎるため、管理されない編集が行われる可能性があります。

代わりに言ってください

「読んで references/reviewer2_comment6_reanalysis_plan.md。承認された編集のみを適用します。範囲外のものがあれば中止してください。」

これにより、エージェントに境界と検証パスが与えられます。
