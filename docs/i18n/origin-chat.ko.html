기준선 고정

유지하다 original_main.tex 변함없이. 작업 main.tex. 수정 지시를 자동으로 생성합니다.

1

응답 논리 초안 작성

원고를 만지기 전에 리뷰어의 실제 관심사를 이해하십시오.

2

references/ 아래 계획 만들기

계획은 허용된 파일, 허용된 위치, 금지된 변경, 근거 및 확인을 정의합니다.

3

인간이 승인하거나 분할함

계획이 너무 광범위하면 분할하세요. 명확하면 실행하십시오.

4

Executor는 계획만 적용합니다.

Codex는 허용된 파일을 편집하고 실행 메모를 기록합니다.

5

PDF 빌드 및 검사

원고와 응답 편지를 편집합니다. 페이지 또는 줄 위치를 요청하기 전에 실제 PDF를 확인하세요.

6

응답 편지 동기화

응답 편지에는 계획되거나 상상된 변경 사항이 아닌 실제 원고 변경 사항이 반영되어야 합니다.

7

커밋 및 푸시

승인된 모든 작업 패키지는 깔끔한 Git 체크포인트가 됩니다.

reviewer comment
→ response logic
→ references/reviewer1_comment1_revision1.md
→ human approval
→ edit main.tex only within the plan
→ compile main.pdf + response_letter.pdf
→ verify PDF locations
→ update response letter
→ generate redline
→ git commit + push

상담원 역할

단일 에이전트가 모든 역할을 수행할 수 있지만 역할이 분리되면 워크플로가 더 명확해집니다.

전략가 에이전트

리뷰어의 고민을 진단하고, 계획을 작성하고, 톤을 조절하고, 작업 분할 여부를 결정합니다. 원고를 직접 편집해서는 안 됩니다.

집행자 에이전트

승인된 계획을 읽고, 파일을 편집하고, LaTeX를 컴파일하고, 레드라인을 생성하고, 수행된 작업을 기록합니다. 과학을 자유롭게 재해석해서는 안 됩니다.

감사 대리인

PDF를 확인하고, 페이지/라인 참조를 확인하고, 응답 편지의 진실성을 확인하고, 편집이 계획을 초과하지 않았는지 확인합니다.

인간 사령관

계획을 승인하고, 작업 세분화를 결정하고, 과학적 방향을 판단하고, 최종 개정을 승인하거나 거부합니다.

You are working on an academic manuscript revision.

First read the relevant plan in references/.
Do not edit anything outside the plan.
If the plan is too broad or vague, stop and propose a smaller plan.
Apply only the approved edits.
After editing, update the plan's Execution Notes.
Run the build command.
Inspect the generated PDF if available.
Update the response letter only with verified changes.
Generate redlined_main.tex when original_main.tex is available.
Show git diff summary.
Commit and push this work package.
