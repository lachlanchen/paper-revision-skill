Die Idee in einem Satz

Lassen Sie einen AI-Agenten nicht gleichzeitig denken, bearbeiten, überprüfen und erklären. Erzwingen Sie zunächst die Begründung in einem Markdown-Plan und lassen Sie dann den Redakteur nur diesen genehmigten Plan ausführen.

1

Denken Sie zuerst nach

Verwenden Sie ein Argumentationsmodell, um die Bedenken des Prüfers zu diagnostizieren und einen Plan zu erstellen.

2

Gate-Änderungen

Kein Plan, keine Manuskriptänderungen. Der Plan ist die Bearbeitungsgrenze.

3

Überprüfen Sie die Wahrheit

Kompilieren Sie PDF, überprüfen Sie es und aktualisieren Sie dann den Antwortbrief mit verifizierten Standorten.

4

Commit-Verlauf

Jedes akzeptierte Arbeitspaket wird zu einem sauberen Git-Schritt, der überprüft oder zurückgesetzt werden kann.

Erstellen Sie vor jeder sinnvollen Bearbeitung einen Markdown-Plan. Legen Sie es darunter references/. Wenn Sie sich später fragen, warum etwas geändert wurde, ist der Grund da. Der Agent kann diese Dateien auch lesen und den Verlauf verstehen.

Meinen Sie eine Markdown-Datei pro Prüferausgabe oder eine Datei, die den gesamten Prozess ständig anhängt?

Chen Miao

Normalerweise viele Dateien. Jede Datei entspricht dem aktuellen Revisionspaket. Zum Beispiel: reviewer1_comment1_revision1.md, figure4_caption_shorten_plan.md, oder main_text_shorten_plan.md.

Die Größe ist dynamisch. Ein Plan kann einen kleinen Absatz, eine Bildunterschrift, einen Rezensentenkommentar oder einen globalen Kürzungsdurchlauf umfassen. Aber Plan und Ausführung sollten eins zu eins übereinstimmen.

Die „Runde“, die Sie meinen, ist also nicht jede kleine Codex-Nachricht. Ist es der Moment, in dem wir Codex tatsächlich ein Arbeitspaket ausführen lassen?

Chen Miao

Ja. Es ist genau das Paket, das Sie Codex übergeben und sagen: Wenden Sie dies an.

Wenn sich der Plan für den Agenten einfach anfühlt, lassen Sie ihn ausführen. Wenn es zu schwierig erscheint, teilen Sie den Plan in kleinere Pläne auf. Anhand der Planqualität lässt sich oft erkennen, ob der Makler Probleme hat.

Praktische Regel: Wenn der Plan vage klingt, wird die Änderung wahrscheinlich vage. Spaltung vor der Ausführung, nicht nach Beschädigung des Manuskripts.

Feifei Robbie

Stammt der Plan von Ihnen, von ChatGPT oder von Codex selbst?

Chen Miao

Jeder von ihnen kann es produzieren, aber der endgültige Plan muss etwas sein, mit dem ich einverstanden bin. Codex ist gut darin, Dateien zu bearbeiten, LaTeX zu reparieren und zu kompilieren. Aber es ist schwächer bei der Entscheidung, warum und wie die Wissenschaft und der Wortlaut überarbeitet werden sollen.

Deshalb frage ich ChatGPT oft nach der Revisionsstrategie, speichere diese als Markdown und lasse sie dann von Codex anwenden.

Wissen: ChatGPT. Technische Ausführung: Codex.

Feifei Robbie

Was passiert, wenn der Plan ausgeführt wird, das Ergebnis jedoch nicht ideal ist und geringfügige Anpassungen erforderlich sind? Erstellen Sie für jede noch so kleine Anpassung eine neue Markdown-Datei?

Chen Miao

Nicht notwendig. Wichtig ist, nicht jede noch so kleine Chat-Nachricht aufzuzeichnen. Wichtig ist, dass der endgültig ausgeführte Plan ein guter Plan ist.

Wenn der Plan nicht gut ist, überarbeiten Sie ihn, überschreiben Sie ihn oder bitten Sie ein anderes Modell, ihn zu verbessern. Aber der Agent sollte durch den endgültigen Plan eingeschränkt werden. Der Kern besteht darin, den Bearbeitungsbereich zu steuern.

C
Chen Miao

Ich habe festgestellt, dass der Agent, wenn er bei einer Aufgabe nicht gut genug ist, anfängt, sie nach dem Zufallsprinzip zu bearbeiten. Je mehr es bearbeitet, desto chaotischer wird es. Aber sobald ich vor der Bearbeitung einen Plan verwende, wird der Prozess kontrollierbar.

Es reduziert die Belastung. Der Agent muss nicht gleichzeitig darüber nachdenken, was er ändern und wie er die Dateien bearbeiten soll.

Feifei Robbie

Sie sind also wie Zhuge Liang und legen die Strategie fest, und Codex geht Zhao Yun ins Spiel?

Chen Miao

Genau. Teile und herrsche. Lassen Sie den Strategen sich auf die Strategie und den Ausführenden auf die Umsetzung konzentrieren.

Endgültige Übersetzung: Der Mensch ist immer noch der Befehlshaber. Der Agent kann planen, bearbeiten, prüfen und kompilieren, sollte aber nicht stillschweigend über die wissenschaftliche Richtung entscheiden.

Der Workflow als Betriebssystem

Die Methode funktioniert, weil sie eine chaotische Revision in eine Folge kleiner, überprüfbarer Transaktionen umwandelt.

0

Frieren Sie die Grundlinie ein

Behalten original_main.tex unverändert. Arbeite daran main.tex. Redline automatisch generieren.

1

Entwerfen Sie die Antwortlogik

Verstehen Sie die wahren Bedenken des Rezensenten, bevor Sie das Manuskript anfassen.

2

Erstellen Sie einen Plan unter references/

Der Plan definiert zulässige Dateien, zulässige Speicherorte, verbotene Änderungen, Gründe und Überprüfung.

3

Der Mensch stimmt zu oder spaltet sich

Wenn der Plan zu umfassend ist, teilen Sie ihn auf. Wenn es klar ist, führen Sie es aus.

4

Der Testamentsvollstrecker wendet nur den Plan an

Codex bearbeitet die zulässigen Dateien und zeichnet Ausführungshinweise auf.

5

Erstellen und prüfen Sie PDF

Erstellen Sie Manuskript und Antwortschreiben. Überprüfen Sie den tatsächlichen PDF, bevor Sie Seiten- oder Zeilenpositionen beanspruchen.

6

Antwortschreiben synchronisieren

Das Antwortschreiben muss tatsächliche Manuskriptänderungen widerspiegeln, nicht geplante oder eingebildete Änderungen.

7

Commit and push

Jedes akzeptierte Arbeitspaket wird zu einem sauberen Git-Checkpoint.

reviewer comment
→ response logic
→ references/reviewer1_comment1_revision1.md
→ human approval
→ edit main.tex only within the plan
→ compile main.pdf + response_letter.pdf
→ verify PDF locations
→ update response letter
→ generate redline
→ git commit + push

Agent roles

Ein einzelner Agent kann alle Rollen übernehmen, der Arbeitsablauf ist jedoch klarer, wenn die Rollen getrennt sind.

Strategist Agent

Diagnostiziert Bedenken des Prüfers, schreibt den Plan, kontrolliert den Ton und entscheidet, ob die Aufgabe aufgeteilt werden soll. Das Manuskript sollte nicht direkt bearbeitet werden.

Executor Agent

Liest den genehmigten Plan, bearbeitet Dateien, kompiliert LaTeX, erstellt Redline und zeichnet auf, was getan wurde. Es sollte die Wissenschaft nicht frei uminterpretieren.

Auditor Agent

Überprüft PDF, verifiziert Seiten-/Zeilenverweise, prüft die Richtigkeit des Antwortschreibens und stellt sicher, dass die Bearbeitung den Plan nicht überschreitet.

Human Commander

Genehmigt den Plan, entscheidet über die Granularität der Aufgabe, beurteilt die wissenschaftliche Ausrichtung und akzeptiert oder lehnt die endgültige Überarbeitung ab.

Zwei Revisionsmodi im Vergleich

Standardanweisung Codex

Diese Eingabeaufforderung verwandelt die Chat-Philosophie in eine ausführbare Agentenanweisung.

You are working on an academic manuscript revision.

First read the relevant plan in references/.
Do not edit anything outside the plan.
If the plan is too broad or vague, stop and propose a smaller plan.
Apply only the approved edits.
After editing, update the plan's Execution Notes.
Run the build command.
Inspect the generated PDF if available.
Update the response letter only with verified changes.
Generate redlined_main.tex when original_main.tex is available.
Show git diff summary.
Commit and push this work package.

Sag es nicht

„Überarbeiten Sie das Papier entsprechend den Kommentaren des Gutachters.“

Dies ist zu weit gefasst und lädt zu unkontrollierter Bearbeitung ein.

Sagen Sie stattdessen

„Lies references/reviewer2_comment6_reanalysis_plan.md. Wenden Sie nur die genehmigten Änderungen an. Hören Sie auf, wenn etwas außerhalb des Rahmens liegt.“

Dadurch erhält der Agent eine Grenze und einen Verifizierungspfad.
