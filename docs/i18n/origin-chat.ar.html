الفكرة في جملة واحدة

لا تدع وكيل AI يفكر ويحرر ويتحقق ويشرح كل ذلك مرة واحدة. قم أولاً بفرض المنطق في خطة Markdown، ثم اسمح لوكيل التحرير بتنفيذ تلك الخطة المعتمدة فقط.

1

فكر أولا

استخدم نموذجًا منطقيًا لتشخيص مخاوف المراجع وكتابة الخطة.

2

تعديلات البوابة

لا توجد خطة ولا تغييرات في المخطوطة. الخطة هي حدود التحرير.

3

التحقق من الحقيقة

قم بتجميع PDF وفحصه، ثم قم بتحديث خطاب الرد بالمواقع التي تم التحقق منها.

4

ارتكاب التاريخ

تصبح كل حزمة عمل مقبولة خطوة Git نظيفة يمكن مراجعتها أو التراجع عنها.

سير العمل كنظام تشغيل

تعمل هذه الطريقة لأنها تحول المراجعة الفوضوية إلى سلسلة من المعاملات الصغيرة التي يمكن التحقق منها.

0

تجميد خط الأساس

احتفظ original_main.tex دون تغيير. العمل على main.tex. إنشاء الخط الأحمر تلقائيًا.

1

قم بصياغة منطق الاستجابة

افهم الاهتمام الحقيقي للمراجع قبل لمس المخطوطة.

2

قم بإنشاء خطة ضمن references/

تحدد الخطة الملفات المسموح بها، والمواقع المسموح بها، والتغييرات المحظورة، والأساس المنطقي، والتحقق.

3

يوافق الإنسان أو ينقسم

إذا كانت الخطة واسعة جدًا، قم بتقسيمها. إذا كان واضحا، تنفيذه.

4

المنفذ يطبق الخطة فقط

يقوم Codex بتحرير الملفات المسموح بها ويسجل ملاحظات التنفيذ.

5

قم ببناء وفحص PDF

تجميع المخطوطة وخطاب الرد. تحقق من PDF الفعلي قبل المطالبة بمواقع الصفحة أو الخط.

6

مزامنة خطاب الرد

يجب أن يعكس خطاب الرد تغييرات حقيقية في المخطوطة، وليس تغييرات مخطط لها أو متخيلة.

7

الالتزام والدفع

تصبح كل حزمة عمل مقبولة نقطة تفتيش Git نظيفة.

reviewer comment
→ response logic
→ references/reviewer1_comment1_revision1.md
→ human approval
→ edit main.tex only within the plan
→ compile main.pdf + response_letter.pdf
→ verify PDF locations
→ update response letter
→ generate redline
→ git commit + push

أدوار الوكيل

يمكن لوكيل واحد أداء جميع الأدوار، ولكن سير العمل يكون أكثر وضوحًا عندما يتم فصل الأدوار.

وكيل استراتيجي

يقوم بتشخيص مخاوف المراجعين، وكتابة الخطة، والتحكم في أسلوب العمل، ويقرر ما إذا كان سيتم تقسيم المهمة أم لا. لا ينبغي أن يقوم بتحرير المخطوطة مباشرة.

وكيل التنفيذ

يقرأ الخطة المعتمدة، ويحرر الملفات، ويجمع LaTeX، وينشئ الخط الأحمر، ويسجل ما تم إنجازه. ولا ينبغي لها أن تعيد تفسير العلم بحرية.

وكيل مراجع الحسابات

التحقق من PDF، والتحقق من مراجع الصفحة/السطر، والتحقق من صدق خطاب الاستجابة، والتأكد من عدم تجاوز التحرير للخطة.

القائد البشري

يوافق على الخطة، ويقرر تفاصيل المهمة، ويحكم على الاتجاه العلمي، ويقبل المراجعة النهائية أو يرفضها.

قائمة مراجعة ملف الخطة

قسم الخطةما ينبغي أن يجيب
تعليق المصدرما تعليق المراجع/المحرر أو الهدف الداخلي الذي أدى إلى حدوث ذلك؟
التشخيصما هو القلق الحقيقي وراء التعليق؟
النطاقما هي الملفات والمواقع التي يمكن تغييرها؟
التغييرات المحظورةما الذي يجب على الوكيل تجنبه صراحة؟
التعديلات المقترحةما هي التغييرات الدقيقة التي ينبغي إجراؤها؟
معايير القبولكيف نعرف أن التعديل قد اكتمل؟
التحققكيف سيتم التحقق من المخطوطة ورسالة الرد والمراجع والخط الأحمر؟

مقارنة وضعين للمراجعة

تعليمات Codex القياسية

تعمل هذه المطالبة على تحويل فلسفة الدردشة إلى تعليمات وكيل قابلة للتنفيذ.

You are working on an academic manuscript revision.

First read the relevant plan in references/.
Do not edit anything outside the plan.
If the plan is too broad or vague, stop and propose a smaller plan.
Apply only the approved edits.
After editing, update the plan's Execution Notes.
Run the build command.
Inspect the generated PDF if available.
Update the response letter only with verified changes.
Generate redlined_main.tex when original_main.tex is available.
Show git diff summary.
Commit and push this work package.

لا تقل

"راجع الورقة وفقًا لتعليقات المراجع."

هذا واسع جدًا ويدعو إلى التحرير غير المنضبط.

قل بدلا من ذلك

"اقرأ references/reviewer2_comment6_reanalysis_plan.md. قم بتطبيق التعديلات المعتمدة فقط. توقف إذا كان أي شيء خارج النطاق ".

وهذا يعطي الوكيل حدودًا ومسارًا للتحقق.
