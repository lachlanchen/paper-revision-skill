Plan-Gated Paper Revision Skill

一個乾淨且輕微戲劇化的起源聊天，解釋瞭如何使 AI 輔助的手稿修訂受控、可追溯且不那麼混亂。

用一句話表達這個想法

不要讓 AI 特務同時思考、編輯、驗證和解釋。 首先將推理強制寫入 Markdown 計劃，然後讓編輯代理僅執行該批准的計劃。

1

先想一想

使用推理模型來診斷審查者的擔憂並編寫計劃。

2

門編輯

沒有計劃，沒有手稿更改。該計劃是編輯邊界。

3

驗證真相

編譯，檢查 PDF，然後使用經過驗證的位置更新回應信。

4

提交歷史

每個接受的工作包都會成為一個乾淨的 Git 步驟，可以進行審查或回滾。

清理原始聊天記錄

這是原始聊天的可讀、翻譯和簡單闡述的版本 Chen Miao 和 Feifei Robbie。無關的歷史玩笑已被刪除。這個有趣的類比之所以被保留，是因為它抓住了架構：戰略家、執行者和人類指揮官。

Chen Miao

我曾經有一個使用 AI 修改論文的工作流程。關鍵是要保持流程的線性，否則整個修改就會變得混亂。

首先，將提交的版本保留為 original_main.tex。工作文件仍然被稱為 main.tex。然後讓agent生成 redlined_main.tex。即使沒有提交紅線，它對於檢查發生了什麼變化也非常有用。

C
R
Feifei Robbie

那麼基準被凍結，當前稿件保持相同的檔名，並且紅線是自動產生的？

Chen Miao

正是如此。然後請AI先起草回覆信。逐一審查者的評論。先了解回應邏輯，然後讓AI制定稿件修改計劃，儲存為Markdown，檢查後才執行。

以這種方式循環瀏覽所有評論。也要寫一個建置腳本，自動編譯最新的稿件、最新的回覆信和紅線版本。

C
設計注意事項： 回信不是事後的想法。這是修訂版的邏輯圖。如果無法對答覆進行辯護，則手稿編輯可能尚未準備好。
Chen Miao

在每次有意義的編輯之前，建立一個 Markdown 計劃。把它放在下面 references/。後來，當你想知道為什麼要改變某些東西時，原因就在那裡。代理還可以讀取這些文件並了解歷史記錄。

沒有計劃，你只有結果，而沒有思考過程。

C
R
Feifei Robbie

您的意思是每個審閱者問題一個 Markdown 文件，還是一個不斷附加整個過程的文件？

Chen Miao

通常有很多文件。每個文件對應於目前的修訂包。例如： reviewer1_comment1_revision1.md, figure4_caption_shorten_plan.md, 或 main_text_shorten_plan.md.

大小是動態的。一個計劃可以涵蓋一個小段落、一個圖形標題、一個審稿人評論或一個全局縮短通行證。但計劃和執行應該是一一對應的。

C
R
Feifei Robbie

所以你所說的「回合」並不是每一個微小的 Codex 訊息。是我們真正讓Codex執行工作包的那一刻嗎？

Chen Miao

是的。這正是您交給 Codex 並說：應用這個的包。

如果代理覺得該計劃很容易，就讓它執行。如果看起來太困難，請將計劃分成更小的計劃。您通常可以從計劃品質判斷代理商是否陷入困境。

C
實用規則： 如果計劃聽起來很模糊，那麼編輯也可能會變得模糊。在執行前分割，而不是在稿件損壞後分割。

Feifei Robbie

該計劃是來自您、來自 ChatGPT 還是來自 Codex 本身？

Chen Miao

他們誰都可以製作，但最終的方案必須是我認可的。 Codex 擅長編輯檔案、修復 LaTeX 和編譯。但它在決定為何以及如何修改科學和措辭方面較弱。

所以我經常向ChatGPT詢問修改策略，將其儲存為Markdown，然後讓Codex應用它。

知識：ChatGPT。技術執行：Codex。

C
R
Feifei Robbie

如果計劃執行了但結果不理想，需要小幅調整怎麼辦？您是否為每一個微小的調整建立一個新的 Markdown 檔案？

Chen Miao

沒有必要。重要的是不要記錄每一個微小的聊天訊息。重要的是，最終執行的計劃是一個好的計劃。

如果計劃不好，就修改它、覆蓋它，或請另一個模型來改進它。但代理人應該受到最終計劃的約束。核心是控制編輯範圍。

C
Chen Miao

我發現，當代理人在某項任務上能力不強時，它就會開始隨機編輯。編輯得越多，就越混亂。但一旦我在編輯前使用了計劃，整個過程就變得可控了。

它減輕了負擔。代理無需同時考慮要更改什麼以及如何編輯文件。

C
R
Feifei Robbie

那你就像Zhuge Liang設定策略一樣，而Codex是Zhao Yun去戰場嗎？

Chen Miao

正是如此。分而治之。讓戰略家專注於策略，讓執行者專注於執行。

最終翻譯： 人類仍然是指揮官。代理人可以策劃、編輯、審核、編譯，但不應該默默地決定科學方向。

作為作業系統的工作流程

這個方法之所以有效，是因為它將混亂的修訂轉換為一系列小型的、可驗證的事務。

0

凍結基線

保留 original_main.tex 不變。繼續努力 main.tex。自動產生紅線。

1

起草響應邏輯

在接觸稿件前先了解審稿者真正關心的問題。

2

在references/下建立計劃

該計劃定義了允許的文件、允許的位置、禁止的更改、理由和驗證。

3

人類批准或分裂

如果計劃太寬泛，請將其拆分。如果清楚了，就執行它。

4

執行者僅應用該計劃

Codex 編輯允許的檔案並記錄執行註解。

5

建置並檢查 PDF

編寫手稿和回覆信。在聲明頁或行位置之前檢查實際的 PDF。

6

同步回覆信

回信必須反映真實的稿件更改，而非計畫或想像的更改。

7

提交並推動

每個接受的工作包都會成為一個乾淨的 Git 檢查點。

reviewer comment
→ response logic
→ references/reviewer1_comment1_revision1.md
→ human approval
→ edit main.tex only within the plan
→ compile main.pdf + response_letter.pdf
→ verify PDF locations
→ update response letter
→ generate redline
→ git commit + push

代理角色

單一代理可以執行所有角色，但當角色分開時，工作流程會更加清晰。

策略代理

診斷審查者的擔憂，編寫計劃，控制語氣，並決定是否拆分任務。不應直接編輯稿件。

執行代理人

讀取核准的計劃，編輯文件，編譯LaTeX，產生紅線，並記錄所做的事情。它不應該隨意重新解釋科學。

審計代理

檢查 PDF，驗證頁面/行引用，檢查回覆信的真實性，並確保編輯沒有超出計畫。

人類指揮官

批准計劃，決定任務粒度，判斷科學方向，接受或拒絕最終修改。

問題→技能機制

常見故障計劃門控解決方案實用規則
代理變動太大該計劃定義了確切的允許文件和位置。若計劃中沒有，請勿編輯。
回覆信誇大其詞僅在 PDF 檢查後新增位置聲明。在手稿實際修改之前，切勿寫「我們修改了」。
修訂歷史變得不清晰每個工作包都有一個計劃和 Git 提交。一項計劃，一項執行，一項提交。
大型任務使代理超載總體計劃在執行前被分割。模糊的計劃→分裂的計劃。
新會話失去上下文的 references/ 目錄充當外部記憶體。在編輯之前讓代理人閱讀相關計劃。

計劃文件清單

計劃部分應該回答什麼
來源評論哪個審稿人/編輯評論或內部目標觸發了此問題？
診斷評論背後真正關心的是什麼？
適用範圍哪些文件和位置可以更改？
禁止的更改代理人應該明確避免什麼？
提議的編輯具體應該做哪些改變？
驗收標準我們如何知道編輯已完成？
驗證如何檢查手稿、回信、參考文獻和紅線？

標準Codex指令

該提示將聊天理念轉變為可執行的代理指令。

You are working on an academic manuscript revision.

First read the relevant plan in references/.
Do not edit anything outside the plan.
If the plan is too broad or vague, stop and propose a smaller plan.
Apply only the approved edits.
After editing, update the plan's Execution Notes.
Run the build command.
Inspect the generated PDF if available.
Update the response letter only with verified changes.
Generate redlined_main.tex when original_main.tex is available.
Show git diff summary.
Commit and push this work package.

不說

“根據審稿人的意見修改論文。”

這太寬泛並且會導致不受控制的編輯。

說代替

「讀 references/reviewer2_comment6_reanalysis_plan.md。僅套用經批准的編輯。如果有任何超出範圍的事情就停下來。 」

這為代理提供了邊界和驗證路徑。
